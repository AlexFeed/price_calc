# Price Calculator

## Описание проекта

`Price Calculator` — это MVP-прототип веб-приложения для автоматизированного расчёта ставок перевозок.

Проект был разработан как прототип решения для компании **АТК Групп**, которая занимается арендой и продажей морских и железнодорожных контейнеров, а также работает с перевозками железнодорожным транспортом.

Основная идея проекта — заменить неудобный и нестабильный расчёт ставок в таблицах на отдельное веб-приложение с понятным интерфейсом, базой данных, возможностью ручной проверки записей и дальнейшей автоматизацией обработки входящих писем.

Проект остался в состоянии **MVP**. В нём реализована базовая рабочая логика калькулятора и управления ставками, но часть функций, связанных с полноценным автоматическим парсингом писем и промышленной эксплуатацией, находится на уровне прототипа.

## Суть решаемой задачи

До разработки отдельного приложения расчёт ставок предполагалось вести через таблицы. Такой подход создаёт несколько проблем:

- таблицы начинают работать медленно при большом объёме данных;
- формулы легко сломать при изменении структуры таблицы;
- пользователю сложно работать с большим количеством записей;
- трудно внедрять автоматическую обработку писем;
- сложно понять, кто и когда изменил данные;
- необработанные письма и некорректные ставки можно легко потерять;
- сохраняется риск ошибок из-за ручного ввода дат, чисел и текстовых значений.

Проект решает задачу:

> создать простой веб-интерфейс для расчёта ставок перевозок, хранения данных, ручной проверки записей и подготовки основы для автоматической обработки писем.

## Основная функциональность MVP

### Калькулятор ставок

Главная функция проекта — расчёт ставки перевозки по выбранным параметрам.

Пользователь задаёт:

- город отправления;
- город назначения;
- тип контейнера;
- вид транспорта.

После этого приложение ищет подходящие записи в базе данных и рассчитывает основную статистику:

- минимальную ставку;
- максимальную ставку;
- среднюю ставку;
- медианную ставку;
- количество найденных записей.

По умолчанию в расчёте используются актуальные ставки за последние 14 дней. Это позволяет не учитывать устаревшие значения при расчёте.

### База ставок

В проекте реализовано хранение ставок в базе данных.

Каждая запись содержит информацию о маршруте, типе контейнера, транспорте, клиенте, валюте, значении ставки, дате, email и статусе обработки.

Ставки можно использовать как для расчёта, так и для ручной проверки и редактирования.

### Управление записями

В приложении есть интерфейс для работы с базой ставок.

Пользователь может:

- просматривать список ставок;
- искать нужные записи;
- фильтровать записи;
- добавлять новые ставки;
- редактировать существующие данные;
- удалять записи;
- отслеживать статус обработки записи.

Это снижает зависимость от ручного ведения таблиц и делает работу с данными более контролируемой.

### Статусы обработки

Для ставок предусмотрены статусы:

- `correct` — корректная запись;
- `incorrect` — некорректная запись;
- `pending` — запись ожидает проверки.

Эта логика нужна для сценария, где данные могут поступать автоматически из писем, но часть записей требует ручной проверки.

### Обработка писем

В репозитории есть отдельный модуль `email-processor`.

Он предназначен для будущей автоматизации обработки входящих писем:

- подключение к почтовому ящику;
- получение новых писем;
- извлечение текста письма;
- определение данных ставки;
- отправка результата в Django-приложение через API.

На этапе MVP эта часть реализована в упрощённом виде. Она показывает общий принцип работы, но не является полноценным парсером писем разных форматов и языков.

### API для приёма ставок

Приложение содержит API-endpoint для добавления ставок из внешнего сервиса.

Этот endpoint может использоваться email processor или другим интеграционным модулем, который передаёт в Django-приложение данные о новой ставке в JSON-формате.

## Логика работы приложения

Общий сценарий работы выглядит так:

1. Пользователь открывает веб-интерфейс калькулятора.
2. Выбирает параметры перевозки.
3. Django-приложение ищет подходящие ставки в базе данных.
4. Система рассчитывает статистику по найденным значениям.
5. Пользователь получает результат расчёта.
6. При необходимости пользователь переходит в интерфейс управления записями и проверяет данные вручную.

Дополнительный сценарий, заложенный в архитектуру MVP:

1. На почту приходит письмо со ставкой.
2. Email processor получает письмо.
3. Из письма извлекаются данные.
4. Данные отправляются в Django API.
5. Новая ставка появляется в базе.
6. Пользователь проверяет корректность записи через интерфейс.


## API проекта

В MVP реализованы несколько API-направлений:

- API калькулятора;
- API управления ставками;
- API пользователей;
- API уведомлений;
- API приёма ставок из внешнего сервиса.

Основной endpoint для приёма ставок:

```text
POST /api/receive_rate/
```

Пример JSON-запроса:

```json
{
  "origin_city": "Москва",
  "destination_city": "Санкт-Петербург",
  "container_type": "20ft",
  "transport_type": "морской",
  "rate": 100.00,
  "input_date": "2025-11-10T12:00:00Z",
  "email": "client@example.com"
}
```

## Как запустить проект локально

### 1. Клонировать репозиторий

```bash
git clone https://github.com/AlexFeed/price_calc.git
cd price_calc
```

### 2. Создать виртуальное окружение

```bash
python -m venv .venv
```

### 3. Активировать виртуальное окружение

Для Windows:

```bash
.venv\Scripts\activate
```

Для Linux/macOS:

```bash
source .venv/bin/activate
```

### 4. Установить зависимости

```bash
pip install -r requirements.txt
```

### 5. Применить миграции

```bash
python manage.py migrate
```

### 6. Загрузить тестовые данные

```bash
python manage.py load_sample_rates
```

### 7. Запустить сервер

```bash
python manage.py runserver
```

После запуска приложение будет доступно по адресу:

```text
http://127.0.0.1:8000/
```

Django Admin:

```text
http://127.0.0.1:8000/admin/
```

## Ограничения MVP

Проект находится на стадии MVP, поэтому часть решений реализована в упрощённом виде:

- используется SQLite;
- приложение рассчитано на локальный запуск;
- email parser работает как прототип;
- API-защита реализована через простой API-key;
- отсутствует полноценная production-конфигурация;
- автоматическое извлечение данных из писем требует дальнейшей доработки.

## Итог

`Price Calculator` — это MVP веб-приложения для автоматизации расчёта и управления ставками перевозок.

Проект демонстрирует переход от ручной работы с таблицами к отдельному Django-приложению, в котором можно хранить ставки, рассчитывать значения по маршруту, проверять записи и закладывать основу для автоматической обработки писем.

Основная ценность проекта — рабочая концепция системы, которая снижает зависимость от ручных таблиц, уменьшает риск ошибок и делает процесс расчёта ставок более прозрачным и управляемым.
